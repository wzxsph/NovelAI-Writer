# NovelAI Writer — 产品需求文档

## Problem Statement

网络文学创作者在撰写30万字以上长篇时，面临严重的上下文割裂问题：人物状态、物品设定、事件逻辑在章节间存在不连贯，导致作者不得不花费大量时间"修bug"而非创作。同时，当前LLM生成的文本缺乏"人味"——幽默感、情绪张力不足，难以达到市场认可的网文水准。现有AI写作工具缺乏针对长篇创作的专门优化，没有成熟的一站式解决方案。

## Evidence

- **Assumption** — 需要通过用户调研和原型内测验证
- 支撑逻辑：网文市场庞大（2023年中国网络文学市场规模超300亿），用户付费意愿强；IP改编产业链成熟；供需矛盾存在
- 核心痛点：状态割裂（人物/物品/事件）+ AI文风问题（缺乏幽默、风趣、动人）

## Users

- **Primary**: 写30万字以上网文的长篇作者（有完本经验或新手均覆盖）
- **Secondary**: 希望快速迭代从灵感到第一版小说的作者
- **Not for**:
  - 短篇、散文、诗歌等其他文学形式
  - 纯手写原教旨主义者（对AI有道德抵触）
  - 协同写作团队（v1聚焦单人创作流）

## Hypothesis

**主假设（一致性）**

我们相信「基于上下文索引的角色/物品/事件状态自动追踪系统」将解决「长篇写作中前后设定割裂、修改成本高」的问题，对于「写30万字以上网文的长篇作者」。当我们看到「内测用户因设定矛盾导致的返工修改时间减少60%以上」时，我们就知道了。

**辅助假设（文风）**

我们相信「通过上下文感知的风格提示词注入」将缓解「AI生成文本缺乏人味和情绪张力」的问题，对于「希望作品更有感染力的网文作者」。当我们看到「至少70%的内测用户认为AI续写在阅读流畅度和情绪传递上接近人工修改水平」时，我们就知道了。

## Scope

**MVP (v1.0)**

- **一致性引擎**：人物、物品、事件状态的自动更新、追踪与冲突提醒
- **上下文感知续写**：基于已有设定与状态文档进行续写或改写，确保每次输出在世界规则内
- **轻量级文风调节**：通过提示词注入幽默、动人等风格，初步让输出更有人味

**Out of scope**

- 一键生成整本书 — 为什么：违背创作规律，MVP聚焦验证一致性引擎价值
- 短篇、散文、诗歌等 — 为什么：聚焦网文长篇
- 多人协同写作 — 为什么：插件架构限制，v1仅支持单人本地创作流（未来可扩展）
- 原创世界观自动生成 — 为什么：用户创建世界观，AI辅助生成草稿，人工审阅后纳入
- 平台发布对接或盈利分析 — 为什么：只解决创作环节，不涉及分发

### Phase Details

**Phase 1: 状态文档架构设计**
- **Goal**: 定义角色/事件/物品三大状态库的数据模型
- **Scope**: 状态数据结构、JSON Schema、版本管理策略
- **Success signal**: 状态模型能覆盖十维一致性需求

**Phase 2: 动态状态追踪引擎**
- **Goal**: 实现续写时自动抽取新角色/物品/事件入状态库
- **Scope**: 状态提取器、增量更新逻辑、冲突检测
- **Success signal**: 状态自动更新覆盖率达到90%以上

**Phase 3: 上下文一致性校验**
- **Goal**: 续写内容与状态库冲突时预警用户
- **Scope**: 十维一致性校验规则、预警机制、用户确认流程
- **Success signal**: 一致性校验通过率>85%

**Phase 4: 逐章续写生成**
- **Goal**: 基于前文 + 状态文档生成单章草稿
- **Scope**: 续写提示词模板、上下文注入策略、输出格式化
- **Success signal**: 生成内容符合状态约束

**Phase 5: 轻量级文风调节**
- **Goal**: 通过提示词注入幽默、动人等风格
- **Scope**: 风格提示词库、用户可调的文风参数
- **Success signal**: 70%以上用户认为文风接近人工修改水平

**Phase 6: 用户交互界面**
- **Goal**: 状态文档查看/编辑/覆写交互
- **Scope**: CLI命令、状态查看、手动覆写机制
- **Success signal**: 用户能直观理解和使用状态系统

**Phase 7: 内测与度量**
- **Goal**: 首批用户完成30万字，验证关键指标
- **Scope**: 内测用户招募、数据收集、指标验证
- **Success signal**: 返工减少60%+，文风满意度70%+

## Decisions Log

| Decision | Choice | Alternatives | Rationale |
|----------|--------|--------------|-----------|
| MVP核心 | 多章节连续写作一致性保障 | 单章高质量、一次生成 | "可控、不出bug"是工具变工具的关键 |
| 一致性 vs 文风优先级 | 一致性优先，文风辅助 | 文风优先 | 一致性可量化验证，文风较主观 |
| 状态管理 | 待定（TBD） | 分布式JSON文件 vs SQLite | JSON文件轻量易读，SQLite结构化查询强；需技术验证后决策 |
| 文风方案 | 轻量级提示词注入 | 完整文风控制系统 | v1验证为主，不做过度工程 |
